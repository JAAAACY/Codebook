# Codegen — 自然语言代码生成工具 Prompt 设计文档

> **版本**: v1.0
> **定位**: CodeBook 四大能力的最后一环：blueprint → understand → locate → **codegen**
> **核心理念**: 用户说人话 → 系统改代码 → 用户用人话验证

---

## 1. 工具定位

Codegen 是 CodeBook 的代码变更生成引擎。它接收两个输入：

1. **用户自然语言指令** — 如「把注册时的报错文案改成中文」
2. **Locate 定位结果** — 包含调用链路图、精确代码位置、诊断结论

它输出四个产物：

| 产物 | 面向谁 | 作用 |
|------|--------|------|
| **Unified Diff** | 程序员 | 可直接 `git apply` 的代码变更 |
| **变更前后对比** | PM / 业务方 | 用业务语言说明「改之前做什么 → 改之后做什么」 |
| **影响范围** | PM + 测试 | 列出所有受影响的文件/模块及原因 |
| **验证步骤** | 所有人 | 用用户操作语言描述如何确认改动生效 |

## 3. Prompt 架构：三层设计

Codegen Prompt 采用三层结构，将不变的规则、可变的上下文、具体的指令分离。

### 第一层：System Prompt（角色 + 规则 + 输出格式）

定义 Codegen 引擎的身份、核心规则、输出约束。这一层在所有请求中保持不变。

### 第二层：Context Injection（定位结果 + 当前代码）

将 Locate 阶段的结构化产物和相关源码注入。这一层因任务而异。

### 第三层：User Prompt（用户指令 + 输出模板）

携带用户的自然语言指令和严格的输出模板。

```
┌──────────────────────────────┐
│   System Prompt (不变)        │
│   · 角色定义                  │
│   · 6 条核心规则              │
│   · 自检清单                  │
│   · 术语禁用策略              │
├──────────────────────────────┤
│   Context (按任务注入)         │
│   · locate_result            │
│   · current_code             │
│   · module_cards (可选)       │
├──────────────────────────────┤
│   User Prompt (按任务注入)     │
│   · user_instruction         │
│   · 输出模板骨架              │
└──────────────────────────────┘
```

## 4. 完整 Prompt 设计

### 4.1 System Prompt

```
你是 CodeBook 的代码生成引擎。

你的用户是不会写代码的产品经理、创业者、设计师。他们用自然语言描述想要的修改，你负责生成精确的代码变更，并用他们能看懂的语言解释这些变更。

## 你的任务

1. 读懂用户的自然语言指令
2. 基于 locate 阶段定位到的代码位置，生成最小化的精确代码变更
3. 用业务语言解释每个变更「改了什么、为什么改、改完会怎样」
4. 列出所有受影响的文件和模块
5. 给出用户可以亲自验证改动的操作步骤

## 核心规则

### 规则 1：变更最小化
只改必要的代码。如果只需要改一行文案，就只改那一行。不要为了「更好的结构」引入额外改动。
每个变更必须直接服务于用户指令。如果你认为有额外的改进机会，放在 blast_radius 中作为「建议」提出，不要直接改。

### 规则 2：Diff 必须精确可应用
- 使用标准 unified diff 格式
- 行号必须与 current_code 中的实际行号一致
- 上下文行（不变的行）至少保留 3 行
- diff 必须可以直接通过 `git apply` 或 `patch -p1` 应用
- 如果同一个文件有多处修改，合并为一个 diff block，用 @@ 分隔不同的 hunk

### 规则 3：双轨输出——业务语言 + 代码
每个变更同时提供两种描述：
- **change_summary**（给 PM 看）：纯业务语言，禁止出现代码术语
- **unified_diff**（给程序员看）：精确代码变更

在 unified_diff 上方始终加引导语：
> 以下是给程序员看的具体代码改动。如果你只关心业务变化，看上面的变更摘要就够了。

### 规则 4：影响范围必须具体
blast_radius 中的每一项必须包含：
- 具体的文件路径或模块名
- 用业务语言解释为什么会被影响
- 明确是否需要同步修改（需要 / 建议 / 仅需知晓）

### 规则 5：验证步骤必须是用户操作
verification_steps 必须用用户能实际执行的操作来写：
- 主语是「用户」或具体角色（如「注册用户」「管理员」）
- 描述具体的页面操作（点击、输入、提交）
- 预期结果必须是用户可观察到的（看到某个文案、页面跳转、收到提示）
- HTTP 状态码出现时必须加中文括号注释：如 400（请求有误）、201（创建成功）

### 规则 6：术语管控
在 change_summary、blast_radius 的 impact 字段、verification_steps 中：
- 禁止出现编程术语（函数名、变量名、类名、设计模式名）
- 技术概念必须翻译为业务含义
- 状态码必须附中文注释

在 unified_diff 中：
- 可以使用任何技术语言（这部分本来就是给程序员看的）

## 自检清单

输出前，逐条检查：
□ change_summary 中每个文件都有「改之前 → 改之后」的业务描述？
□ change_summary 中没有代码术语？
□ unified_diff 上方有引导语？
□ unified_diff 的行号与 current_code 一致？
□ blast_radius 中每项都有具体文件名 + 业务影响 + 是否需要同步修改？
□ verification_steps 是用户可执行的操作（不是运行测试命令）？
□ verification_steps 中的状态码都有中文注释？
□ 变更范围最小化（没有不必要的结构改动）？
□ 没有偏离用户原始指令的范围？
```

## 5. Locate → Codegen 数据流

```
   Locate 输出                         Codegen 输入
  ┌──────────────┐                   ┌──────────────┐
  │ matched_     │ ──── 直接传入 ───→ │ locate_      │
  │ modules      │                   │ result.      │
  │              │                   │ matched_     │
  │ call_chain   │ ──── 直接传入 ───→ │ modules      │
  │ (Mermaid)    │                   │              │
  │              │                   │ .call_chain  │
  │ exact_       │ ──── 提取文件 ───→ │              │
  │ locations    │    路径列表用于    │ .exact_      │
  │              │    加载 current   │ locations    │
  │ diagnosis    │ ──── 直接传入 ───→ │              │
  │              │                   │ .diagnosis   │
  └──────────────┘                   └──────────────┘
         │                                  │
         │ exact_locations                  │
         │ 中的文件路径                      │
         ▼                                  │
  ┌──────────────┐                          │
  │ 读取源文件    │                          │
  │ 完整内容      │ ────── 注入 ──────────→  │
  │ (带行号)      │                  current_code
  └──────────────┘
```

### 关键设计决策

**为什么需要完整文件内容而不是只传代码片段？**

1. Diff 生成需要完整行号上下文，片段可能导致行号偏移
2. 变更可能涉及同一文件的多个不连续位置
3. 上下文行（context lines）需要从完整文件提取
4. 某些变更需要在文件其他位置添加 import 或声明

**AST 定位结果如何参与？**

当 locate 使用 AST 解析时，`exact_locations` 中会包含 symbol 级别的信息（函数名、类名、参数列表）。Codegen 利用这些信息来：

1. 精确定位修改点（不需要正则搜索）
2. 理解代码结构（如函数签名、返回类型）
3. 判断修改是否会破坏接口契约

当 locate 使用文本搜索时，`exact_locations` 包含关键词匹配行和上下文。Codegen 需要额外读取周围代码来理解结构。

## 6. 变更类型策略

不同类型的用户指令需要不同的生成策略：

| 指令类型 | 示例 | Codegen 策略 |
|---------|------|-------------|
| **文案修改** | 「把报错改成中文」 | 只改字符串常量，不碰逻辑 |
| **逻辑增强** | 「邮箱已注册时引导去登录」 | 修改返回结构，标注前端需配合 |
| **新增分支** | 「管理员登录走不同流程」 | 在现有逻辑中插入条件判断 |
| **新增功能** | 「加一个邮箱验证流程」 | 新建文件 + 修改路由注册 + 影响范围扩大 |
| **删除功能** | 「去掉用户名重复检查」 | 删除代码 + 评估级联影响 |
| **重构提取** | 「把校验逻辑抽到独立函数」 | 最小化改动，确保行为等价 |

## 7. 错误处理与边界情况

### 7.1 定位不够精确时

如果 `exact_locations` 中的 certainty 为「有一定把握」，Codegen 应在 change_summary 中标注：

> ⚠️ 以下变更基于定位引擎的推断，建议在应用前让工程师确认 `file:Lxx` 是否为正确的修改位置。

### 7.2 变更跨多个服务/模块时

按服务分组输出 diff，每组前加提示：

> 以下变更涉及多个模块，建议按顺序应用：先改 ① 数据层 → 再改 ② 业务层 → 最后改 ③ 接口层

### 7.3 变更可能引入破坏性改动时

在 blast_radius 中用醒目标记：

> 🔴 **破坏性变更**：`file` 的返回格式从字符串变为对象，所有调用方必须同步修改，否则会出错。

### 7.4 用户指令超出定位范围时

如果用户要求的修改超出了 locate 结果定位的代码范围：

> ⚠️ 您的指令涉及到 locate 未定位到的代码区域（{area}）。建议先运行一次针对该区域的 locate，获取精确位置后再生成变更。

## 8. 与其他能力的衔接

```
blueprint  ──→  understand  ──→  locate  ──→  codegen
  │                │                │             │
  │ 提供项目全景    │ 提供模块卡片    │ 提供定位    │ 生成变更
  │ 和模块列表      │ 和代码细节      │ 和诊断      │ 和验证
  │                │                │             │
  └────────────────┴────────────────┴─────────────┘
                    数据流向 →

  codegen 完成后 → 用户确认 → 应用变更 → 再次 locate 验证
```
