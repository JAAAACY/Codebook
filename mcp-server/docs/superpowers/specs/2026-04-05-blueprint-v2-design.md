# Codebook Blueprint v2 — UE5 蓝图式交互设计规格

## 立项初衷

Codebook 的目标是「让不会写代码的人也能理解软件项目」。Blueprint v1 偏离了这个目标——输出是开发者视角的模块堆叠和技术统计，对 PM、领域专家、CEO 不具备可读性。

Blueprint v2 回归初衷：用业务语言讲故事，用可视化画布替代列表堆叠，参照 UE5 蓝图的双击下钻体验。

## 目标用户

- PM：理解项目做什么、各能力间怎么协作、哪里有风险
- 领域专家：验证代码实现是否匹配业务需求
- CEO/投资人：快速了解项目全景

## 核心体验

两级画布视图 + 常驻 LLM 对话框：

```
Overview（世界地图）──双击节点──→ 子页面（模块内部逻辑）
         │                                  │
         └────── 右侧 LLM 对话框 ───────────┘
```

---

## Overview 页（世界地图）

### 布局

全屏画布，可缩放、可拖拽。

右上角：项目名 + 一句话定位（如「Claw Code 是一个 AI 编程助手的核心引擎」）。

右侧：可收起的 LLM 对话框。

### 节点

每个节点代表一个**业务功能模块**（不是代码目录）。

节点内容：
- 中文业务名（大字，如「插件系统」）
- 一句话说明（小字，如「管理第三方扩展的加载与执行」）
- 健康度色标：绿色边框（健康）/ 黄色（需关注）/ 红色（风险）
- 左侧输入引脚 + 右侧输出引脚（连线锚点）

### 连线

节点间的连线表示数据/控制流向。

连线上标注动词（如「调用」「传递消息」「返回结果」「读取配置」）。

### 布局规则

按业务主线从左到右排列，核心流程一眼可见。入口在最左，输出在最右。

### 交互

- 缩放/拖拽画布
- 悬停节点：无额外效果（信息已在节点上）
- **双击节点**：进入该模块的子页面
- 点击连线：无效果（信息已在连线标注上）
- 右侧对话框随时可用

---

## 子页面（模块内部逻辑）

### 进入方式

从 Overview 双击某个业务功能节点进入。

### 导航

左上角面包屑：`Overview / 插件系统`，点击 Overview 可返回。

### 布局

同样是全屏画布，可缩放、可拖拽。

### 函数级节点（默认视图）

每个节点代表一个函数/方法。节点内容分三层：

```
┌───────────────────────────────────┐
│ 验证插件签名                        │  ← 业务说明（做什么）
│───────────────────────────────────│
│ check_signature()                  │  ← 函数名
│ plugins/verify.rs:42               │  ← 文件位置
│ 读取插件包的签名证书，与可信列表      │  ← 实现逻辑解释（怎么做）
│ 比对，不匹配则拒绝加载并记录日志      │
│───────────────────────────────────│
│ ● plugin_path      → ● is_valid   │  ← 输入/输出引脚
└───────────────────────────────────┘
```

- **顶部**：中文业务说明（做什么）—— 大字高亮
- **中部**：函数名 + 文件位置 + 实现逻辑解释（怎么做）—— 业务语言描述实现步骤
- **底部**：输入参数引脚（左）和输出/返回引脚（右）

### 连线

节点间连线表示调用关系。引脚对引脚连接。

连线上可标注数据类型或业务含义。

### 折叠（步骤级视图）

密集区域支持折叠：多个相关函数节点折叠为一个步骤级大节点。

折叠节点显示：
- 步骤名（如「权限验证流程」）
- 包含的函数数量
- 整体输入/输出引脚

双击折叠节点可展开回函数级。

### 批注

每个节点右上角有批注图标。点击可添加文字批注，批注持久保存在蓝图文件中。

## LLM 对话框

### 位置

右侧常驻面板，可收起/展开。贯穿 Overview 和子页面。

### 上下文感知

对话框知道用户当前在看哪个视图（Overview 还是某个子页面）、最后交互的节点是什么。

### 功能

1. **回答问题**：「这个模块为什么标红？」「如果移除这个函数会影响什么？」
2. **解释逻辑**：「插件加载的完整流程是什么？」
3. **发起修改提案**：
   - PM 在对话中表达修改意图（如「权限检查应该区分管理员和普通用户」）
   - LLM 先用业务语言解释变更影响（如「流程会从 A → B 变为 A → 判断角色 → B1/B2」）
   - PM 确认方向
   - 弹出提案面板

### 提案面板

对话确认后弹出的独立面板：
- **左侧**：业务逻辑变更说明（流程 before/after 对比）
- **右侧**：代码变更预览（unified diff）
- **底部**：「批准 / 修改 / 拒绝」三个操作按钮

## 信息生成策略

### 业务名称 + 说明 + 逻辑解释

扫描流程新增 LLM 摘要步骤：

1. Codebook 完成代码解析（现有 Step 1-4）
2. 将每个模块的代码结构（函数签名、类名、依赖关系、关键代码片段）发送给 LLM
3. LLM 返回：
   - 模块的业务功能名（中文）
   - 一句话说明
   - 每个函数的业务说明 + 实现逻辑解释
   - 模块间连线的动词标注
4. 结果缓存到 ProjectMemory，增量扫描时只更新变更的部分

### 降级方案

LLM 不可用时，回退到基于规则的推断：
- 目录名/文件名关键词映射（如 `auth` → 「认证」，`plugins` → 「插件」）
- 函数名驼峰/下划线拆分转中文
- 质量低于 LLM 但保证可用

## 技术约束

### 自包含 HTML

生成的蓝图必须是单个 HTML 文件，双击即可在浏览器打开。所有 JS/CSS 内联。

### 画布引擎

纯 HTML/SVG/Canvas 实现，参考 React Flow 的设计理念但不依赖 React。需要支持：
- 节点拖拽和画布缩放/平移
- 节点间连线（贝塞尔曲线 + 引脚）
- 双击下钻和面包屑返回
- 折叠/展开节点组

### 对话框

蓝图 HTML 内的对话框通过 MCP 协议与宿主 LLM 交互。独立打开蓝图时对话框显示「连接 MCP 后可用」的占位提示。

### 批注存储

批注数据嵌入 HTML 文件的 `<script>` 标签中（JSON），修改后通过浏览器 File API 或导出功能保存。

---

## 不做什么（v2 范围外）

- 多人协作 / 实时同步
- 版本对比（两次扫描的 diff）
- 代码编辑器（只预览 diff，不做在线编辑）
- 移动端适配
